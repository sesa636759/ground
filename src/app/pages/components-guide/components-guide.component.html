When to Use Components

Prefer a Ground component when you need consistent behavior, design-token alignment, and accessibility support out of the box. Build custom wrappers only for product-specific behavior.

Use directly

Buttons, inputs, overlays, and navigation controls with standard interaction patterns.

Wrap lightly

Add domain defaults, labels, and validation while keeping base component APIs intact.

Avoid forking

Do not copy component internals unless there is a strict platform-level requirement.

Import Pattern

Import only the components you render to keep bundles smaller and code easier to review.

import { Component, CUSTOM_ELEMENTS_SCHEMA } from '@angular/core';

@Component({
  selector: 'app-invoice-toolbar',
  standalone: true,
  schemas: [CUSTOM_ELEMENTS_SCHEMA],
  template: `
    <ui-button variant="primary">Save</ui-button>
    <ui-button variant="outline">Preview</ui-button>
  `,
})
export class InvoiceToolbarComponent {}

Composition Rules

Compose components around user workflows. Keep state in containers and keep presentational components stateless whenever possible.

  • Keep one primary action per view and de-emphasize secondary actions.
  • Group related fields in cards or panels, not arbitrary nested layouts.
  • Use feedback components (snackbar, dialog) only after user intent is clear.

Accessibility Checklist

Before shipping, verify these checks in each page that uses components:

  • Every interactive element is keyboard reachable.
  • Focus order matches visual order.
  • Labels, helper text, and errors are announced by assistive tech.
  • Contrast and touch targets meet your baseline accessibility criteria.
